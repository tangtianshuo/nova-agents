## Typography

**Maximum 4 font sizes, 2 font weights** (per design checker constraint):

| Role | Size | Weight | Line Height | CSS Variable |
|------|------|--------|-------------|--------------|
| Page title | 28px | 600 (semibold) | 1.2 | `--text-3xl` |
| Section heading | 18px | 600 (semibold) | 1.4 | `--text-lg` |
| Body | 16px | 400 (normal) | 1.6 | `--text-base` |
| Button/Label | 13px | 500 (medium) | 1.5 | `--text-sm` |

**Usage in Phase 4**:
- Dialog title: `--text-lg` (18px, semibold) — dialog header
- Form label: `--text-sm` (14px, medium) — field labels
- Form input: `--text-sm` (14px) — user input
- Button text: `--text-sm` (14px, medium) — primary/secondary buttons
- Helper text: `--text-xs` (12px) — descriptions, hints

**Weight hierarchy** (2 weights + button exception):
- Weight 600 (semibold): Dialog title, section heading, card titles, emphasized labels
- Weight 400 (normal): Body text, descriptions
- Weight 500 (medium): Button text, form labels — exception per design system spec

**Source**: `--text-*` tokens in `src/renderer/index.css` lines 259-267. Consistent with Phase 3 contract.

## Visual Focal Point

**Primary visual anchor in dialogs**: The dialog title (e.g., "添加自定义供应商") at the top-left establishes context. The close button (X) at top-right provides exit. The primary action button ("保存") at bottom-right draws attention for completion. Required field indicators (asterisk *) in red guide the eye to mandatory fields.

**Primary visual anchor in GeneralSection**: Section title ("通用设置") at top-left. Toggle switches draw attention as interactive controls. Theme selector buttons use active state styling to show current selection.

## Design Decisions

### Decision 4-1: Dialog Component Granularity

**Decision:** Create separate dialog components (CustomProviderDialog, CustomMcpDialog, DeleteConfirmDialog) instead of one unified dialog.

**Rationale:**
- Each dialog has distinct form structure and validation rules
- Reusability: DeleteConfirmDialog can be used by both providers and MCPs
- Maintainability: Smaller components are easier to understand and modify
- Testing: Easier to test individual dialog logic

### Decision 4-2: Dual Mode for CustomMcpDialog

**Decision:** Support both form-based input and JSON paste in CustomMcpDialog.

**Rationale:**
- JSON mode is power-user friendly (copy-paste from documentation)
- Form mode is user-friendly (GUI with validation)
- Toggle button allows easy switching between modes
- Reduces friction for advanced users while maintaining accessibility

### Decision 4-3: Inline Confirmation vs DeleteConfirmDialog

**Decision:** Phase 03 used inline delete confirmation in ProviderCard. Phase 04 introduces DeleteConfirmDialog for consistency.

**Rationale:**
- Inline confirmation is good for simple cases (ProviderCard)
- But complex deletes (with cascading effects) need full dialog
- DeleteConfirmDialog can show more context (dependencies, impact)
- Reusable across different entity types (providers, MCPs, workspaces)

**Implementation:**
- Keep inline confirmation in ProviderCard (Phase 03 pattern)
- Use DeleteConfirmDialog for general deletion cases
- Document when to use each approach

### Decision 4-4: GeneralSection Layout

**Decision:** Use card-based layout for GeneralSection (same as other sections).

**Rationale:**
- Consistent visual design with ProviderSection and McpSection
- Each setting group (startup, theme, workspace) gets its own card
- Easier to scan and understand than one long form
- Cards can be reordered or expanded in future
