# nova-agents 战略审查报告

> 日期：2026-04-08
> 版本：v0.1.x (开发中)
> 审核范围：UI/UX · 产品定位 · 商业化路径

---

## 一、现状概览

**项目阶段**：功能基本完善，已实现多 Tab Chat、MCP 工具、IM Bot 集成、定时任务、内嵌终端、用户 SMS 认证（Phase 2 开发中）。商业化基础设施**尚未构建**。

**技术评级**：架构设计扎实（Session-Centric Sidecar、双运行时、Plugin Bridge），代码质量高，技术债务低。

**商业化评级**：产品能力完备，变现基础设施为零。

## 二、UI/UX 审查

### 2.1 设计系统评估

**优点**

| 维度 | 评价 |
|------|------|
| 设计语言 | 温暖纸张质感，差异化鲜明，阅读体验优化到位 |
| Color Token | Ink/Paper/Accent/Semantic 四层体系完整 |
| Typography | 16px 正文基准，H1-H6 阶梯清晰，跨平台字体策略完善 |
| 组件规范 | 按钮/卡片/输入框/标签/Overlay 均有详细规范 |
| 动效规范 | Transition duration/easing 体系完整 |

**设计系统覆盖率估算：~75%**

**具体问题**

1. **Settings 页面体积异常**：单文件超过 350KB，渲染性能风险高，需拆分或懒加载。
2. **设计规范执行无强制手段**：CLAUDE.md 明确禁止硬编码颜色（`#fff`、`bg-blue-500`），但无 ESLint rule 或 CI 检查，违规代码可随时混入。
3. **登录/注册页 UI 尚未对齐设计系统**：Phase 02-02 刚完成，但 LoginPage/RegisterPage 是否使用了 CSS Token 而非硬编码颜色，需专项审查。
4. **`--hover-bg` vs `--paper-inset` 混用风险**：设计规范区分了列表行 hover（`--hover-bg`）和小按钮 hover（`--paper-inset`），但两类场景边界在代码中容易被错误使用。

**改进建议**

- **立即行动**：为硬编码颜色添加 ESLint rule（`eslint-plugin-no-unstyled-color-literals` 或自定义 rule），防止 Design Token 侵蚀。
- **立即行动**：Settings 页面代码分割，按 Provider/MCP/Agent/Skills 等模块拆为独立 Chunk。
- **短期**：UI-SPEC.md 审查 LoginPage/RegisterPage/OtpInput 三个新组件是否符合 design_guide.md。
- **建议**：参考 [Tailwind CSS Custom-linting 方案](https://tailwindcss.com/docs/customizing-layers#creating-a-custom-rule) 为 Design Token 添加 Stylelint 检查。

### 2.2 关键页面审查

#### Launcher 页面

- **布局**：左右 60/40 分栏，品牌展示清晰，功能入口分区合理。
- **问题**：快捷功能卡片 hover 上浮动效（`translate-y-[-1px]`）规范来源不明（来自 design_guide v2.2.0 更新），需确认已落地。
- **建议**：增加空状态设计（首次安装无工作区时的引导流程）。

#### Chat 页面

- **优势**：多 Tab、Terminal Panel、工具调用卡片、权限请求块等 AI 交互组件规范完整。
- **问题**：长对话列表的滚动性能、思考过程块的折叠逻辑需实测验证。
- **建议**：Chat 消息虚拟化列表（react-virtuoso 已引入）是否启用？如未启用，需验证大数据量场景下的帧率。

#### Settings 页面

- **问题**：代码体积过大（353KB），耦合度高，单一职责原则未遵守。
- **重构建议**：
  ```
  Settings/
    ├── SettingsSidebar.tsx      # 侧边栏导航
    ├── ProviderSettings/        # 供应商配置（懒加载）
    │   ├── ProviderList.tsx
    │   └── ProviderEditPanel.tsx
    ├── McpSettings/           # MCP 工具（懒加载）
    │   ├── McpList.tsx
    │   └── McpInstallDialog.tsx
    ├── AgentSettings/          # Agent 配置（懒加载）
    ├── SkillSettings/          # 技能配置（懒加载）
    └── AccountSettings/       # 账户与订阅（新增商业化模块）
  ```

#### 登录/注册页面

- **现状**：SMS 验证码流程已完成，包含 OtpInput 组件和 60s 倒计时。
- **缺失**：登录后没有"绑定已有账户"路径（用户可能忘记已注册手机号），无 OAuth 第三方登录入口。

## 三、产品定位审查

### 3.1 核心价值主张

**现有定位**：
> "让每个人都有一个懂你、能替你干活的桌面 AI Agent"

**定位分析**：

| 维度 | 评估 |
|------|------|
| 清晰度 | 定位"桌面端 AI 助手"，与 Claude Code（CLI）形成 GUI/CLI 互补 |
| 差异化 | IM Bot 集成 + 本地数据持久化 + 桌面 GUI，差异化明确 |
| 情感连接 | "懂你"、"搭档"语言有温度，但产品中未充分体现（记忆系统、用户画像缺失）|
| 可执行性 | 功能描述清晰，但"懂你"需要记忆/个性化系统支撑 |

### 3.2 目标用户

文档列举：学生、内容创作者、教育工作者、产品经理、开发者。

**问题**：用户分层不清晰。免费用户 vs 付费用户 vs 企业用户的使用场景未区分，产品路线混杂。

**建议**：明确定义 3 个用户象限：

| 象限 | 用户 | 核心需求 | 付费意愿 |
|------|------|----------|----------|
| **个人爱好者** | 开发者、学生 | 体验 AI 能力，隐私敏感 | 低 |
| **专业用户** | 内容创作者、PM | 效率提升、工作流自动化 | 中 |
| **企业团队** | 开发团队、运营团队 | 协作、权限管理、合规 | 高 |

### 3.3 竞争态势

| 竞品 | 定位 | nova-agents 优势 | nova-agents 劣势 |
|------|------|------------------|------------------|
| Claude Code | CLI 开发者工具 | GUI + IM Bot + 多会话 | 品牌认知度低 |
| Cursor | IDE 类 AI 助手 | 通用 Agent + MCP 生态 | 功能深度不如 Cursor |
| Copilot Desktop | Microsoft 生态 | 跨平台 + IM 集成 | 企业背景弱 |
| 任记 | AI 笔记 | 本地优先 | 功能差异大 |

**核心机会**：隐私敏感的专业用户（不希望数据上云）+ 需要 IM 自动化的工作流（现有产品空白）。

## 四、商业化路径审查（核心）

### 4.1 当前商业化状态

**现状：纯基础设施，无产品变现。**

- Anthropic/DeepSeek/Moonshot 等 Provider 费用：用户直接支付给第三方，nova-agents 零抽成。
- 用户认证系统（SMS）已建，但**没有任何与付费绑定的内容**。
- `SubscriptionInfo` 类型仅关联 Anthropic 订阅信息，不是 nova-agents 自己的产品订阅。

### 4.2 缺失的商业化基础设施

```
❌ 无订阅层级（Free/Pro/Enterprise）
❌ 无功能 gating（所有功能对所有用户开放）
❌ 无用量追踪（无法按 API 消耗或工具调用次数计费）
❌ 无支付渠道（无 Stripe/支付宝/微信支付接入）
❌ 无企业计费（无Seat-based 授权、SSO、审计日志）
❌ 无插件/技能市场收入
❌ 无多设备同步订阅（数据绑定设备，用户留存弱）
```

### 4.3 商业化建议路径

#### Phase 1：基础付费层（0→1 变现）

**建议订阅模型**

| 层级 | 价格（参考）| 功能 |
|------|-------------|------|
| **Free** | $0 | 单工作区、单 Provider、MCP×3、基础 IM Bot |
| **Pro** | ¥30/月或$5/月 | 5个工作区、多 Provider、无限 MCP、优先 Sidecar 资源 |
| **Pro+** | ¥60/月或$10/月 | Pro + 高级插件市场、团队同步（3设备）、优先客服 |
| **Enterprise** | 定制 | 私有部署、SSO、审计日志、专属 MCP、自定义模型 |

**功能 gating 矩阵建议**

| 功能 | Free | Pro | Pro+ | Enterprise |
|------|------|-----|------|------------|
| 工作区数 | 1 | 5 | 无限 | 无限 |
| MCP 工具数 | 3 | 无限 | 无限 | 无限 |
| 多 Provider | ❌ | ✅ | ✅ | ✅ |
| IM Bot (Telegram/DingTalk) | ✅ | ✅ | ✅ | ✅ |
| OpenClaw 插件 | ❌ | 部分 | ✅ | ✅ |
| 团队协作/多设备 | ❌ | ❌ | 3设备 | 无限 |
| 企业 SSO | ❌ | ❌ | ❌ | ✅ |
| 优先 Sidecar 资源 | ❌ | ❌ | ✅ | ✅ |
| 插件市场收入分成 | ❌ | ❌ | ✅ | ✅ |

**实施优先级**

1. **立即**：扩展 `SubscriptionInfo` 类型，支持 nova-agents 产品订阅层级（free/pro/enterprise）。
2. **立即**：在 AuthContext 中增加订阅层级状态，UI 根据层级条件渲染功能入口。
3. **短期**：支付渠道接入（建议 Stripe + 国内支付宝/微信支付双轨）。
4. **短期**：用量追踪基础设施（工具调用次数、API Token 消耗、工作区数等）。

#### Phase 2：生态收入（ Marketplace）

| 方向 | 说明 | 优先级 |
|------|------|--------|
| **MCP 工具市场** | 开发者发布付费 MCP 工具，平台抽成 30% | 高 |
| **技能（Skills）市场** | 垂直场景技能包（PM 工作流、客服话术等） | 中 |
| **插件市场** | OpenClaw 插件付费墙 | 低（依赖插件 Bridge 成熟度） |

#### Phase 3：企业级产品

| 方向 | 说明 |
|------|------|
| **私有部署** | 企业内网部署，按 Seat 收费 |
| **SSO/OIDC** | 企业身份提供商集成 |
| **审计日志** | 合规需求，管理员控制台 |
| **私有 MCP 仓库** | 企业内部工具共享 |

### 4.4 垂直产品策略执行问题

`docs/VERTICAL_PRODUCT_STRATEGY.md` 文档了 nova-pm / nova-ecommerce / nova-creator 三个垂直方向，但**代码中完全没有落地**。

**建议**：
- 垂直产品应该是 **Pro/Enterprise 层级的场景化包装**，而非独立产品线。
- 避免重复造轮子，先完成基础付费层，再基于用户反馈决定垂直方向投入。

## 五、数据与留存

### 5.1 关键缺失

| 缺失项 | 影响 |
|--------|------|
| **无用户行为分析** | 不了解用户如何使用产品，无法优化 |
| **无崩溃报告系统** | 生产问题发现滞后 |
| **无功能埋点** | 无法衡量功能价值 |
| **无 A/B 测试基础设施** | 无法迭代优化 |

**建议**：接入分析工具（建议自托管的 Plausible/Umami 或轻量端，保留数据主权）。

### 5.2 用户留存风险

| 风险点 | 描述 |
|--------|------|
| **数据局限于单设备** | 用户换设备后无法同步，工作区丢失 |
| **无个性化记忆** | "懂你"的承诺需要长期记忆系统，当前为空白 |
| **无社交/协作** | 纯工具定位，用户粘性弱 |

**建议**：多设备同步应作为 **Pro+ 层核心功能**推进，这是强留存抓手。

## 六、合规与法律

### 6.1 当前状态

- 项目声明 Apache-2.0 开源许可证
- 接入第三方 API（Anthropic/DeepSeek 等），需用户自行承担 API 费用
- SMS 认证服务由 `@nova-intelligent/auth-sdk` 提供，隐私合规责任归属外部服务

### 6.2 待处理项

| 项 | 建议 |
|----|------|
| **隐私政策** | 桌面应用虽不收集数据，但需告知第三方 API 调用的数据流向 |
| **数据处理协议（DPA）** | 企业客户要求签署 DPA，需提前准备 |
| **出口管制** | AI Agent 涉及跨境数据传输，需评估适用法规 |

## 八、总结

### 核心结论

1. **技术基础扎实**：架构设计优秀，代码质量高，UI 设计系统完整。
2. **商业化完全空白**：产品能力已经 ready，但没有任何变现基础设施。
3. **最大机会**：隐私优先的桌面 AI Agent + 多设备同步 + IM 自动化工作流。
4. **最大风险**：持续投入开发但不建立收入来源，项目可持续性存疑。

### Top 3 优先行动

| # | 行动 | 理由 |
|---|------|------|
| 1 | **扩展 SubscriptionInfo + 功能 Gating** | 商业化基础设施核心，1-2 周可完成 |
| 2 | **Settings 页面重构** | 降低维护成本，为商业化功能腾出架构空间 |
| 3 | **接入支付渠道** | 实现第一步真实收入，建议 Stripe（全球）+支付宝（国内） |
